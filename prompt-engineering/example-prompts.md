# 9-1 예시 프롬프트 모음

## 프롬프트 엔지니어링: 코덱스에게 일 잘 맡기는 요청 구조

## 사용 방법

이 자료는 9-1 강의에서 화면에 띄워놓고 함께 읽거나, 수강생에게 실습용 핸드아웃으로 제공하기 위한 예시 프롬프트 모음입니다.

모든 예시는 아래 6요소를 기준으로 작성했습니다.

- 목표: 무엇을 만들거나 고치고 싶은가
- 컨텍스트: 어떤 파일, 화면, 오류, 상황을 봐야 하는가
- 범위/제약: 어디까지 해도 되고 무엇은 하지 말아야 하는가
- 작업 모드: 설명, 계획, 실행 중 무엇을 원하는가
- 검증: 작업이 잘 끝났는지 어떻게 확인할 것인가
- 출력 형식: 결과를 어떤 형식으로 정리할 것인가

## 0. 나쁜 요청을 좋은 요청으로 바꾸기

### 나쁜 요청

```text
이 앱을 더 예쁘게 만들어주세요.
```

### 좋은 요청

```text
@src/App.jsx 와 @src/App.css 를 기준으로 홈 화면의 첫 인상을 개선하고 싶습니다.

작업 모드:
- 아직 코드는 수정하지 마세요.
- 먼저 개선 계획만 제안해주세요.

목표:
- 제목, 설명, 버튼 영역의 시각적 위계를 더 명확하게 만들고 싶습니다.
- 기존 미니멀한 분위기는 유지해주세요.

범위와 제약:
- 수정 후보 파일은 `src/App.jsx`, `src/App.css`로 제한해주세요.
- 새 라이브러리는 설치하지 마세요.
- 전체 레이아웃 구조를 크게 바꾸지 마세요.
- 모바일에서 텍스트나 버튼이 겹치지 않게 고려해주세요.

계획 형식:
1. 현재 문제
2. 수정 후보
3. 구현 순서
4. 확인 방법
5. 남은 위험
```

## 1. 설명만 받고 싶을 때

### 현재 파일 구조 설명

```text
@src/App.jsx 파일을 초보자 눈높이로 설명해주세요.

작업 모드:
- 설명만 해주세요.
- 코드는 수정하지 마세요.

설명에 포함할 것:
1. 이 파일의 역할
2. 화면에 보이는 요소가 만들어지는 순서
3. 중요한 변수나 함수의 의미
4. 초보자가 헷갈릴 수 있는 부분
5. 다음에 읽으면 좋은 파일
```

### 선택한 코드 설명

```text
선택한 코드가 어떤 역할을 하는지 설명해주세요.

조건:
- 문법 설명보다 실제 실행 흐름을 중심으로 설명해주세요.
- 이 코드가 화면이나 데이터에 어떤 영향을 주는지 알려주세요.
- 코드는 수정하지 마세요.

출력 형식:
1. 한 문장 요약
2. 단계별 실행 흐름
3. 초보자 주의점
```

## 2. 구현 전에 계획만 받고 싶을 때

### Plan Mode용 기본 프롬프트

```text
@src/App.jsx 와 @src/App.css 를 기준으로 메인 페이지의 사용성을 개선하고 싶습니다.

작업 모드:
- 아직 코드는 수정하지 마세요.
- 먼저 구현 계획만 작성해주세요.

목표:
- 첫 화면에서 사용자가 무엇을 해야 하는지 더 빨리 이해하게 만들고 싶습니다.
- 기존 프로젝트 구조는 유지하고 싶습니다.

범위와 제약:
- 수정 후보 파일은 `src/App.jsx`, `src/App.css`로 제한해주세요.
- 새 라이브러리는 설치하지 마세요.
- 데이터 구조나 라우팅 구조는 바꾸지 마세요.

계획 형식:
1. 현재 화면의 문제
2. 수정할 파일과 이유
3. 구체적인 변경 제안
4. 구현 순서
5. 검증 방법
6. 작업 전 확인 질문
```

### 큰 요구사항을 작은 작업으로 나누기

```text
이 프로젝트에 "일기 작성, 목록 보기, 삭제" 기능을 추가하고 싶습니다.

작업 모드:
- 바로 구현하지 말고 작업을 작은 단계로 나눠주세요.

컨텍스트:
- 현재는 정적 HTML/CSS/JS 프로젝트입니다.
- 백엔드나 데이터베이스는 사용하지 않습니다.

제약:
- 브라우저 localStorage만 사용해주세요.
- 새 패키지는 설치하지 않는 방향으로 계획해주세요.
- 입문자가 따라올 수 있게 단계별로 나눠주세요.

출력 형식:
1. 전체 목표
2. 1단계에서 할 일
3. 2단계에서 할 일
4. 3단계에서 할 일
5. 각 단계별 확인 방법
6. 너무 어렵거나 나중으로 미룰 기능
```

## 3. 승인한 계획을 실행시킬 때

### 계획 실행 요청

```text
좋습니다. 방금 제안한 계획 중 1단계만 구현해주세요.

범위와 제약:
- 계획에서 언급한 파일만 수정해주세요.
- 새 라이브러리는 설치하지 마세요.
- 기존 기능이 사라지지 않게 해주세요.

검증:
- 가능하면 기존 실행 명령이나 lint 명령을 확인해서 실행해주세요.
- 실행하지 못한 검증이 있으면 이유를 말해주세요.

작업 후 요약 형식:
1. 변경한 파일
2. 변경한 이유
3. 실행한 검증
4. 브라우저에서 확인할 부분
5. 남은 위험
```

### 일부만 구현하도록 좁히기

```text
방금 계획 중 "문구 개선"과 "버튼 간격 정리"만 구현해주세요.

이번에는 하지 않을 것:
- 컴포넌트 분리
- 새 상태 관리 추가
- 새 라이브러리 설치
- 전체 디자인 재작성

작업 후에는 어떤 변경을 의도적으로 하지 않았는지도 함께 요약해주세요.
```

## 4. 버그를 고치기 전에 원인만 찾고 싶을 때

### 루트커즈 분석 요청

```text
버튼을 눌러도 저장이 되지 않는 문제가 있습니다.

작업 모드:
- 아직 코드는 수정하지 마세요.
- 먼저 원인을 조사하고 수정 방향만 제안해주세요.

컨텍스트:
- 관련 파일은 @index.html, @script.js, @styles.css 입니다.
- 브라우저 콘솔에는 특별한 오류가 보이지 않습니다.

분석에 포함할 것:
1. 문제가 발생할 수 있는 코드 경로
2. 실제 원인으로 보이는 부분
3. 왜 그렇게 판단했는지
4. 가장 작은 수정 방향
5. 수정 후 확인 방법
```

### 오류 메시지 해석 요청

```text
아래 오류 메시지가 무엇을 의미하는지 설명해주세요.

오류:
[여기에 오류 메시지를 붙여넣습니다.]

작업 모드:
- 설명과 원인 추정만 해주세요.
- 파일 수정은 하지 마세요.

출력 형식:
1. 오류의 의미
2. 가장 가능성 높은 원인
3. 확인해야 할 파일
4. 수정 전에 확인할 질문
```

## 5. 리팩터링을 맡길 때

### 작은 리팩터링 요청

```text
@script.js 파일의 중복된 DOM 선택과 이벤트 처리 코드를 조금 정리하고 싶습니다.

작업 모드:
- 실제로 코드를 수정해주세요.

목표:
- 동작은 그대로 유지하면서 읽기 쉽게 정리하는 것이 목표입니다.

범위와 제약:
- `script.js`만 수정해주세요.
- HTML 구조와 CSS 클래스명은 바꾸지 마세요.
- 새 라이브러리는 설치하지 마세요.
- 기능 추가는 하지 마세요.

검증:
- 저장, 삭제, 목록 렌더링이 기존처럼 동작하는지 확인할 방법을 알려주세요.

작업 후 요약:
1. 바꾼 구조
2. 유지한 동작
3. 직접 확인해야 할 기능
4. 리팩터링으로 생길 수 있는 위험
```

### 리팩터링 계획만 요청

```text
@src/components 폴더를 보고 컴포넌트 구조를 더 읽기 쉽게 정리할 수 있을지 검토해주세요.

작업 모드:
- 아직 코드는 수정하지 마세요.
- 리팩터링 계획만 제안해주세요.

제약:
- 파일 이동이나 이름 변경은 최소화하고 싶습니다.
- 기존 import 경로가 많이 깨지는 방식은 피해주세요.
- 입문자 강의용 프로젝트라 너무 추상적인 구조는 피해주세요.

출력 형식:
1. 현재 구조 요약
2. 정리하면 좋은 부분
3. 지금은 건드리지 않는 것이 좋은 부분
4. 추천 작업 순서
5. 각 단계의 위험도
```

## 6. 테스트와 검증을 함께 요청할 때

### 수정 후 검증까지 요청

```text
@src/App.jsx 에서 빈 상태 메시지가 더 친절하게 보이도록 문구를 개선해주세요.

범위와 제약:
- `src/App.jsx`만 수정해주세요.
- 스타일은 바꾸지 마세요.

검증:
- 가능한 경우 프로젝트의 test, lint, build 명령을 확인해서 실행해주세요.
- 실행한 명령과 결과를 요약해주세요.
- 실행하지 못한 검증은 이유를 알려주세요.

작업 후 요약 형식:
1. 변경한 문구
2. 변경 이유
3. 실행한 검증
4. 실행하지 못한 검증
5. 사람이 직접 확인할 화면
```

### 검증 명령만 찾아달라고 요청

```text
이 프로젝트에서 수정 후 어떤 검증 명령을 실행해야 하는지 찾아주세요.

작업 모드:
- 아직 코드는 수정하지 마세요.
- package.json, README, 설정 파일을 보고 검증 루틴만 정리해주세요.

출력 형식:
1. 실행 가능한 명령
2. 각 명령이 확인하는 것
3. 초보자가 먼저 실행하면 좋은 순서
4. 실행 전에 필요한 준비
```

## 7. 결과 요약을 잘 받기 위한 프롬프트

### 작업 요약 요청

```text
방금 작업한 내용을 강의 수강생도 이해할 수 있게 요약해주세요.

출력 형식:
1. 이번 작업의 목표
2. 변경한 파일
3. 파일별 변경 내용
4. 실행한 검증
5. 직접 확인해야 할 화면
6. 남은 위험
7. 다음에 이어서 하면 좋은 작업
```

### diff 리뷰 요청

```text
방금 변경된 diff를 리뷰해주세요.

관점:
- 요청 범위를 벗어난 변경이 있는지
- 초보자용으로 너무 복잡해진 부분이 있는지
- 기능이 깨질 위험이 있는지
- 검증이 부족한 부분이 있는지

출력 형식:
1. 바로 고쳐야 할 문제
2. 지금은 괜찮지만 주의할 점
3. 잘 유지된 부분
4. 추가로 확인할 명령 또는 화면
```

## 8. 프롬프트 자체를 개선해달라고 요청할 때

### 내 프롬프트 점검

```text
아래 프롬프트를 코덱스에게 더 잘 전달되도록 고쳐주세요.

원본 프롬프트:
[여기에 내가 쓴 프롬프트를 붙여넣습니다.]

개선 기준:
- 목표가 분명한지
- 필요한 컨텍스트가 있는지
- 수정 범위와 금지 사항이 있는지
- 설명, 계획, 실행 중 작업 모드가 분명한지
- 검증 기준이 있는지
- 결과 요약 형식이 있는지

출력 형식:
1. 원본 프롬프트의 문제점
2. 개선된 프롬프트
3. 왜 이렇게 바꿨는지
```

## 9. 피해야 할 프롬프트와 안전한 대체안

| 피해야 할 요청 | 왜 위험한가 | 안전한 대체안 |
| --- | --- | --- |
| 이 프로젝트 전체를 알아서 고쳐줘 | 범위가 너무 넓고 의도하지 않은 파일이 바뀔 수 있음 | 먼저 프로젝트 구조를 읽고 3단계 개선 계획만 제안해주세요 |
| 완벽하게 만들어줘 | 완료 기준이 없음 | 이 화면에서 사용자가 다음 행동을 이해하기 쉽게 문구와 버튼 위계를 개선해주세요 |
| 최신 라이브러리로 다 갈아엎어줘 | 의존성, 구조, 강의 난이도가 급격히 커짐 | 새 라이브러리 없이 현재 구조 안에서 가장 작은 개선안을 제안해주세요 |
| 테스트도 알아서 해줘 | 어떤 검증을 했는지 불명확함 | 실행한 검증 명령과 실행하지 못한 검증을 나눠서 요약해주세요 |
| 일단 수정해봐 | 설명/계획/실행 경계가 없음 | 아직 수정하지 말고 먼저 수정 계획과 위험을 알려주세요 |

## 10. 9-1 수강생 미션용 최종 템플릿

아래 템플릿의 대괄호 부분을 바꿔서 사용합니다.

```text
@[파일1] 과 @[파일2] 를 기준으로 [화면/기능]을 개선하고 싶습니다.

작업 모드:
- [설명만 / 계획만 / 실제 수정] 해주세요.
- [아직 코드는 수정하지 마세요 / 승인한 계획대로 구현해주세요].

목표:
- [사용자에게 어떤 경험을 주고 싶은지]
- [이번 작업에서 가장 중요한 결과]

범위와 제약:
- 수정 후보 파일은 [파일 목록]으로 제한해주세요.
- 새 라이브러리는 설치하지 마세요.
- [바꾸면 안 되는 구조나 기능]은 유지해주세요.

검증:
- 가능한 경우 [lint/test/build/브라우저 확인]을 확인해주세요.
- 실행하지 못한 검증은 이유를 말해주세요.

작업 후 출력 형식:
1. 변경한 파일 또는 수정 계획
```
