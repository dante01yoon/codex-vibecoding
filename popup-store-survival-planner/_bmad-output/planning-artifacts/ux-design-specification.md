# UX Design Specification: Pop-up Store Survival Planner

**작성자:** Dante  
**날짜:** 2026-05-18  
**문서 목적:** 입문자 실습용 React 단일 페이지 앱을 만들기 위한 UX 설계 기준을 정의한다.

## 1. Executive Summary

### 1.1 제품 비전

Pop-up Store Survival Planner는 팝업스토어 방문자가 방문 전 준비를 빠르게 정리하고, 현장에서 바로 볼 수 있는 **오늘의 생존 플랜 카드**를 만드는 웹앱이다.

이 UX는 많은 입력을 요구하지 않는다. 사용자는 날짜와 시간대, 대기 시간, 사고 싶은 굿즈, 체크리스트를 짧게 입력하거나 선택한다. 앱은 그 정보를 한 화면에서 요약해 "오늘 무엇을 챙기고, 무엇을 먼저 살지"를 바로 보여준다.

### 1.2 UX 목표

- 사용자가 한 화면 안에서 방문 준비 흐름을 이해한다.
- 텍스트 입력보다 선택, 체크, 짧은 숫자 입력을 우선한다.
- 결과는 항상 생존 플랜 카드에서 즉시 확인된다.
- 모바일 화면에서도 손가락으로 편하게 입력하고 체크할 수 있다.
- React 입문자가 상태 관리, 리스트 렌더링, 조건부 표시, localStorage 저장을 연습할 수 있다.

### 1.3 첫 구현 버전의 핵심 기능

첫 구현 버전은 기능을 4개 흐름으로 제한한다.

1. **방문 정보 입력** — 날짜, 시간대, 예상 대기 시간을 입력한다.
2. **굿즈 예산 정리** — 굿즈 이름, 우선순위, 예상 금액을 최대 5개까지 입력한다.
3. **현장 체크리스트 확인** — 기본 체크리스트를 체크하고 준비 상태를 본다.
4. **생존 플랜 카드와 최근 플랜 저장** — 입력 내용을 카드로 요약하고 localStorage에 최근 플랜 1개를 저장한다.

방문 후 메모, 체크리스트 항목 직접 추가/삭제, 이미지 저장, SNS 공유는 첫 구현 버전에서 제외하거나 선택 과제로 둔다.

## 2. UX Direction

### 2.1 설계 원칙

- **한 화면에서 끝내기:** 페이지 이동 없이 입력과 결과를 함께 보여준다.
- **입력보다 선택:** 시간대와 우선순위는 버튼 또는 select로 고르게 한다.
- **결과 먼저 보이기:** 생존 플랜 카드는 비어 있어도 기본 상태로 항상 보인다.
- **초보자 친화적 구조:** 화면 영역은 React 컴포넌트로 나누기 쉬운 단위로 설계한다.
- **모바일 우선:** 375px 폭에서도 가로 스크롤 없이 사용할 수 있어야 한다.

### 2.2 제외할 UX

- 로그인 화면을 만들지 않는다.
- 결제 화면이나 결제 버튼을 만들지 않는다.
- 지도, 길찾기, 위치 권한 흐름을 만들지 않는다.
- 실시간 재고 상태를 보여주지 않는다.
- SNS 공유 버튼을 만들지 않는다.
- 서버 저장, 동기화, 계정 설정 화면을 만들지 않는다.

## 3. Information Architecture

### 3.1 단일 페이지 구조

앱은 하나의 페이지 안에 아래 순서로 구성한다.

1. **상단 제목 영역**
   - 앱 이름.
   - 짧은 설명.
   - 저장 상태 텍스트.

2. **방문 정보 입력 영역**
   - 방문 날짜.
   - 방문 시간대.
   - 예상 대기 시간.

3. **굿즈 우선순위와 예산 영역**
   - 굿즈 항목 리스트.
   - 항목 추가 버튼.
   - 예산 합계.

4. **현장 체크리스트 영역**
   - 기본 체크리스트.
   - 체크 완료 개수.

5. **오늘의 생존 플랜 카드 영역**
   - 방문 요약.
   - 긴 대기 대비 메시지.
   - 우선 구매 굿즈.
   - 체크리스트 진행 상태.

6. **최근 플랜 저장 영역**
   - 최근 저장 시간.
   - 새로고침 후 복원 안내.
   - 초기화 버튼.

### 3.2 데스크톱 레이아웃

1024px 이상에서는 2열 레이아웃을 사용한다.

- 왼쪽 60%: 입력 영역.
- 오른쪽 40%: 생존 플랜 카드와 최근 플랜 저장 영역.
- 오른쪽 영역은 화면 상단에 가까이 고정하되, sticky가 복잡하면 일반 배치로 둔다.

### 3.3 모바일 레이아웃

768px 미만에서는 1열 레이아웃을 사용한다.

- 상단 제목.
- 방문 정보.
- 굿즈 예산.
- 체크리스트.
- 생존 플랜 카드.
- 최근 플랜 저장.

모바일에서는 생존 플랜 카드가 너무 아래로 밀릴 수 있으므로, 방문 정보 입력 직후 간단 요약 줄을 보여줘도 된다.

## 4. Core User Experience

### 4.1 핵심 경험

사용자는 "팝업스토어 가기 전에 뭘 준비해야 하지?"라는 막연한 걱정을 앱에 입력한다. 앱은 그 걱정을 방문 정보, 굿즈 예산, 체크리스트, 생존 플랜 카드라는 작은 단계로 나누어 보여준다.

가장 중요한 순간은 사용자가 생존 플랜 카드를 보고 "이 정도면 준비됐다"라고 느끼는 순간이다.

### 4.2 사용자 흐름

#### Flow 1: 방문 전 플랜 만들기

1. 사용자가 방문 날짜를 선택한다.
2. 시간대를 선택한다.
3. 예상 대기 시간을 입력한다.
4. 굿즈 항목과 예산을 입력한다.
5. 체크리스트를 확인한다.
6. 생존 플랜 카드가 자동으로 완성된다.

#### Flow 2: 현장에서 체크리스트 확인하기

1. 사용자가 앱을 다시 연다.
2. localStorage에서 최근 플랜이 복원된다.
3. 사용자가 체크리스트 항목을 체크한다.
4. 생존 플랜 카드의 준비 상태가 업데이트된다.

#### Flow 3: 최근 플랜 초기화하기

1. 사용자가 최근 플랜 저장 영역에서 초기화 버튼을 누른다.
2. 앱은 확인 메시지를 보여준다.
3. 사용자가 확인하면 입력값과 localStorage가 초기화된다.

### 4.3 성공 상태

- 사용자는 생존 플랜 카드에서 방문 시간, 예상 대기 시간, 예산 합계, 우선 구매 굿즈, 준비 상태를 한 번에 본다.
- 새로고침 후에도 최근 플랜이 유지된다.
- 체크리스트를 모두 체크하면 준비 완료 메시지가 표시된다.

## 5. Screen Specifications

### 5.1 상단 제목 영역

**목적:** 앱의 역할을 즉시 알려주고 저장 상태를 보여준다.

**요소:**

- 앱 이름: `Pop-up Store Survival Planner`
- 설명 문구: `방문 전 준비를 한 장의 생존 플랜 카드로 정리하세요.`
- 저장 상태: `최근 플랜 저장됨`, `저장 중`, `저장 실패`

**UX 규칙:**

- 랜딩 페이지처럼 과한 hero 영역을 만들지 않는다.
- 첫 화면 안에 방문 정보 입력 영역이 함께 보여야 한다.
- 저장 상태는 작은 텍스트 또는 badge로 표시한다.

### 5.2 방문 정보 입력 영역

**목적:** 생존 플랜의 기본 정보를 빠르게 입력한다.

**요소:**

- 날짜 입력: `input type="date"`
- 시간대 선택: 오전, 오후, 저녁 segmented control 또는 select
- 예상 대기 시간: 숫자 입력, 단위는 분

**상태:**

- 기본 상태: 입력 전에는 생존 플랜 카드에 `날짜 미정`, `시간대 미정` 표시.
- 유효 상태: 입력값이 카드에 즉시 반영.
- 오류 상태: 대기 시간이 음수이거나 숫자가 아니면 `0 이상의 숫자로 입력해주세요.` 표시.

**브라우저 검증 기준:**

- 날짜를 선택하면 카드의 방문 날짜가 바뀐다.
- 시간대를 선택하면 카드의 시간대가 바뀐다.
- 대기 시간에 `150`을 입력하면 긴 대기 대비 메시지가 표시된다.
- 대기 시간에 음수를 입력하면 오류 문구가 보인다.

### 5.3 굿즈 우선순위와 예산 입력 영역

**목적:** 사용자가 사고 싶은 굿즈와 예산을 작게 정리한다.

**요소:**

- 굿즈 이름 입력.
- 우선순위 선택: 높음, 보통, 낮음.
- 예상 금액 입력.
- 항목 추가 버튼.
- 항목 삭제 버튼.
- 예산 합계.

**범위 제한:**

- 굿즈 항목은 최대 5개까지만 허용한다.
- 예산 입력은 0 이상의 숫자만 허용한다.
- 실제 결제나 가격 비교는 제공하지 않는다.

**브라우저 검증 기준:**

- 굿즈 이름과 예상 금액을 입력하면 리스트에 항목이 추가된다.
- 예산 합계가 입력한 금액의 합으로 바뀐다.
- 우선순위가 높은 항목이 생존 플랜 카드에서 먼저 보인다.
- 5개를 초과해 추가하려 하면 더 이상 추가되지 않거나 안내 문구가 보인다.

### 5.4 현장 체크리스트 영역

**목적:** 방문 당일 챙길 것을 체크한다.

**기본 항목:**

- 보조 배터리.
- 물.
- 입장 정보 확인.
- 카드/현금 준비 확인.
- 굿즈 보관 가방.

**첫 구현 범위:**

- 기본 항목 체크/해제만 필수로 구현한다.
- 사용자 정의 항목 추가/삭제는 선택 과제로 둔다.

**브라우저 검증 기준:**

- 체크박스를 클릭하면 체크 상태가 바뀐다.
- 체크한 개수가 진행 상태에 반영된다.
- 모든 항목을 체크하면 생존 플랜 카드에 준비 완료 메시지가 표시된다.
- 새로고침 후 체크 상태가 유지된다.

### 5.5 오늘의 생존 플랜 카드 결과 영역

**목적:** 사용자가 입력한 정보를 읽기 쉬운 카드로 요약한다.

**카드 구성:**

- 카드 제목: `오늘의 생존 플랜`
- 방문 날짜와 시간대.
- 예상 대기 시간.
- 대기 시간 메시지.
- 예산 합계.
- 우선 구매 굿즈 1-3개.
- 체크리스트 진행 상태.

**상황별 메시지 예시:**

- 대기 시간 0-59분: `가볍게 다녀오기 좋은 플랜이에요.`
- 대기 시간 60-119분: `대기가 조금 길 수 있어요. 물과 배터리를 챙겨요.`
- 대기 시간 120분 이상: `긴 대기 예상! 보조 배터리와 물은 꼭 챙겨요.`
- 체크리스트 완료: `준비 완료. 이제 현장에서 당황할 일이 줄었어요.`

**브라우저 검증 기준:**

- 입력을 바꾸면 카드 내용이 새로고침 없이 즉시 바뀐다.
- 입력값이 비어 있어도 카드 레이아웃이 깨지지 않는다.
- 모바일 화면에서 카드가 화면 밖으로 넘치지 않는다.

### 5.6 최근 플랜 저장 영역

**목적:** localStorage 저장과 복원을 사용자가 이해하게 한다.

**요소:**

- 저장 상태 문구.
- 최근 저장 시간.
- `최근 플랜 초기화` 버튼.

**UX 규칙:**

- 여러 플랜 목록은 만들지 않는다.
- 최근 플랜 1개만 저장한다.
- 초기화 버튼은 위험도가 있으므로 보조 버튼 스타일로 둔다.
- 초기화 전에는 `저장된 플랜을 지울까요?` 확인을 거친다.

**브라우저 검증 기준:**

- 입력 후 새로고침하면 최근 플랜이 복원된다.
- 초기화하면 입력값과 카드가 기본 상태로 돌아간다.
- 초기화 후 새로고침해도 이전 플랜이 복원되지 않는다.

## 6. Visual Design Foundation

### 6.1 디자인 방향

`ui-ux-pro-max` 검색 결과는 모바일 우선, 미니멀 단일 컬럼, 높은 대비, 부드러운 카드 UI를 추천했다. 이 프로젝트에는 그중 **작은 도구형 앱에 맞는 부분만** 적용한다.

최종 방향:

- 밝고 가벼운 도구형 UI.
- 과한 장식보다 명확한 입력과 결과.
- 부드러운 카드 구분.
- 모바일에서 손가락으로 누르기 쉬운 컨트롤.
- 브랜드 랜딩처럼 보이지 않는 실제 사용 화면.

### 6.2 Color System

주요 색상은 한 가지 계열로만 치우치지 않게 구성한다.

- Background: `#F8FAFC`
- Surface: `#FFFFFF`
- Primary: `#C0265D`
- Accent: `#D97706`
- Success: `#0F766E`
- Info: `#2563EB`
- Text: `#111827`
- Muted Text: `#475569`
- Border: `#E2E8F0`
- Error: `#DC2626`

**사용 규칙:**

- Primary는 주요 버튼과 활성 상태에만 사용한다.
- Accent는 예산 합계나 중요한 하이라이트에 제한적으로 사용한다.
- Success는 체크리스트 완료 상태에 사용한다.
- Error는 입력 오류에만 사용한다.

### 6.3 Typography

- 기본 폰트: Inter 또는 시스템 sans-serif.
- H1: 28-32px.
- H2: 20-24px.
- 섹션 제목: 16-18px.
- 본문: 14-16px.
- 보조 설명: 13-14px.

**규칙:**

- viewport 폭에 따라 글자 크기를 자동 확대하지 않는다.
- 버튼 안 텍스트가 줄바꿈되어도 높이가 안정적으로 유지되게 한다.
- 강의용 코드에서는 폰트 설정을 복잡하게 만들지 않는다.

### 6.4 Spacing and Shape

- 전체 페이지 여백: 모바일 16px, 태블릿 이상 24-32px.
- 섹션 간격: 20-24px.
- 카드 내부 여백: 16-20px.
- 카드 radius: 8px.
- 입력 높이: 최소 40px.
- 터치 대상: 최소 44px.

## 7. Component Strategy

### 7.1 기본 컴포넌트

- `AppHeader`
- `VisitInfoForm`
- `GoodsBudgetList`
- `ChecklistSection`
- `SurvivalPlanCard`
- `RecentPlanStorage`

### 7.2 재사용 UI 컴포넌트

- `SectionPanel` — 각 입력 영역을 감싸는 단순 패널.
- `TextInput` — label, input, helper text, error text를 함께 가진 입력.
- `NumberInput` — 숫자 입력과 단위 표시.
- `SegmentedControl` — 오전/오후/저녁 선택.
- `PrioritySelect` — 높음/보통/낮음 선택.
- `ChecklistItem` — 체크박스와 라벨.
- `StatusBadge` — 저장 상태, 준비 완료 상태 표시.
- `IconButton` — 삭제, 초기화 같은 짧은 액션.

### 7.3 아이콘 규칙

- 아이콘이 필요하면 Lucide 계열을 사용한다.
- 텍스트 대신 의미가 명확한 아이콘을 쓸 수 있다.
- 삭제는 trash 아이콘, 초기화는 rotate/refresh 아이콘, 저장 상태는 check/cloud 계열 아이콘을 고려한다.
- UI 아이콘으로 이모지를 사용하지 않는다.

## 8. UX Patterns

### 8.1 입력 패턴

- 모든 입력에는 label을 붙인다.
- placeholder는 예시로만 사용하고 label을 대신하지 않는다.
- 숫자 입력에는 `inputmode="numeric"`을 고려한다.
- 대기 시간과 예산은 음수 입력을 허용하지 않는다.
- 오류는 입력 바로 아래에 짧게 표시한다.

### 8.2 선택 패턴

- 시간대는 segmented control을 우선한다.
- 우선순위는 select 또는 작은 radio group으로 구현한다.
- 선택 상태는 색상만으로 구분하지 않고 테두리, 굵기, 텍스트도 함께 사용한다.

### 8.3 체크리스트 패턴

- 체크박스와 라벨 전체를 클릭 가능하게 한다.
- 체크된 항목은 텍스트 색을 약하게 하고 취소선을 선택적으로 사용할 수 있다.
- 완료율은 `3/5 준비됨`처럼 숫자로 보여준다.

### 8.4 피드백 패턴

- 저장 성공: `최근 플랜 저장됨`.
- 저장 실패: `이 브라우저에서는 저장을 사용할 수 없어요.`
- 입력 오류: `0 이상의 숫자로 입력해주세요.`
- 준비 완료: `준비 완료. 현장에서 바로 확인하세요.`

### 8.5 초기화 패턴

- 초기화는 주 버튼이 아니다.
- 초기화 버튼은 최근 플랜 저장 영역 하단에 둔다.
- 클릭 시 브라우저 confirm 또는 간단한 확인 UI를 사용한다.

## 9. Responsive and Accessibility

### 9.1 반응형 기준

- 375px: 모바일 기본 검증 폭.
- 768px: 태블릿 또는 작은 노트북.
- 1024px: 2열 레이아웃 시작 후보.
- 1440px: 최대 너비 제한 확인.

### 9.2 모바일 규칙

- 모든 섹션은 세로로 쌓인다.
- 입력 필드가 화면 밖으로 넘치지 않는다.
- 버튼은 최소 44px 높이를 유지한다.
- 굿즈 항목 하나가 너무 길면 줄바꿈되며 레이아웃을 깨지 않는다.

### 9.3 접근성 규칙

- 모든 input은 label과 연결한다.
- 체크박스는 키보드로 조작 가능해야 한다.
- focus 상태가 눈에 보여야 한다.
- 오류 문구는 색상만으로 전달하지 않는다.
- 저장 상태 변화는 화면에 텍스트로 표시한다.
- 애니메이션은 150-300ms 안에서 짧게 사용한다.

## 10. Browser Acceptance Criteria

브라우저에서 첫 구현 버전을 검증할 때 아래 항목을 확인한다.

1. 날짜를 선택하면 생존 플랜 카드의 날짜가 즉시 바뀐다.
2. 시간대를 선택하면 생존 플랜 카드의 시간대가 즉시 바뀐다.
3. 예상 대기 시간 120분 이상을 입력하면 긴 대기 대비 메시지가 보인다.
4. 굿즈 2개와 예산을 입력하면 예산 합계가 정확히 표시된다.
5. 우선순위가 높은 굿즈가 생존 플랜 카드에서 먼저 보인다.
6. 체크리스트 항목을 체크하면 준비 상태 숫자가 바뀐다.
7. 모든 체크리스트 항목을 체크하면 준비 완료 메시지가 보인다.
8. 새로고침 후 방문 정보, 굿즈, 체크 상태가 복원된다.
9. 최근 플랜 초기화를 실행하면 입력값과 저장값이 사라진다.
10. 모바일 375px 폭에서 가로 스크롤 없이 사용할 수 있다.

## 11. Implementation Notes for Beginner React

이 섹션은 UX 요구사항을 React 실습 단위로 연결하기 위한 참고다.

### 11.1 추천 상태 구조

- `visitDate`
- `visitTimeSlot`
- `waitMinutes`
- `goodsItems`
- `checklistItems`
- `lastSavedAt`

방문 후 메모는 첫 구현 버전에서 제외하고, 시간이 남으면 선택 과제로 추가한다.

### 11.2 추천 구현 순서

1. 정적 레이아웃 만들기.
2. 방문 정보 입력과 카드 반영.
3. 굿즈 리스트 추가와 예산 합계.
4. 체크리스트 체크 상태.
5. localStorage 저장과 복원.
6. 초기화 기능.
7. 모바일 레이아웃 다듬기.

### 11.3 선택 과제

- 체크리스트 항목 직접 추가/삭제.
- 방문 후 한 줄 메모.
- 카드 텍스트 복사.
- 간단한 theme toggle.

## 12. Open UX Questions

1. 첫 강의 버전에서 체크리스트 항목 직접 추가/삭제를 뺄지, 선택 과제로 둘지 결정해야 한다.
2. 방문 후 메모를 첫 구현에 넣을지, 후속 과제로 뺄지 결정해야 한다.
3. 시간대 선택을 segmented control로 할지 select로 단순화할지 결정해야 한다.
4. 생존 플랜 카드가 모바일에서 입력 영역보다 먼저 한 번 보여야 하는지 확인해야 한다.

## 13. Final UX Decision

첫 구현 UX는 **"입력 폼이 많은 앱"이 아니라 "방문 준비 체크 도구"처럼 느껴져야 한다.** 사용자는 위에서 아래로 준비를 채우고, 오른쪽 또는 아래의 생존 플랜 카드에서 결과를 바로 확인한다.

이 UX spec의 기준 구현은 다음을 만족하면 완료로 본다.

- 단일 페이지.
- 핵심 기능 4개 흐름.
- 최근 플랜 1개 localStorage 저장.
- 모바일 사용 가능.
- 로그인, 결제, 지도, 재고, SNS, 서버 없음.
