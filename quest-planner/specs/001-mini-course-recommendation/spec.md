# Feature Specification: 미니 코스 추천

**Feature Branch**: `001-mini-course-recommendation`

**Created**: 2026-05-17

**Status**: Draft

**Input**: User description: "Weekend Quest Planner의 첫 번째 MVP로, 사용자가 오늘의 기분, 사용 가능 시간, 예산, 동행 여부를 선택하면 조건에 맞는 미니 코스를 추천한다. 추천은 샘플 데이터 기반이며, 최근 추천 결과는 같은 브라우저에서 다시 볼 수 있어야 한다. 외부 서비스, 로그인, 서버 기능은 제외한다."

## Clarifications

### Session 2026-05-17

- Q: 조건에 맞는 코스가 여러 개 있을 때 어떤 기준으로 대표 추천 1개를 고를까요? → A: 조건을 모두 만족하는 코스 중 총비용이 가장 낮은 코스, 동률이면 총시간이 짧은 코스를 추천한다.
- Q: 최근 추천 결과에는 어떤 정보를 저장하고 보여줘야 하나요? → A: 최근 1건의 선택 조건, 코스 요약, 활동별 시간과 비용을 저장하고 새 추천 시 교체한다.
- Q: 입력 선택지는 MVP에서 어느 정도까지 확정할까요? → A: 기분 4개, 시간 3개, 예산 3개, 동행 2개를 고정 선택지로 확정한다.
- Q: 조건에 맞는 추천이 없을 때 빈 상태는 어느 정도로 안내해야 하나요? → A: "정확히 맞는 코스 없음"을 보여주고 시간 또는 예산을 넓혀보라고 안내한다.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 조건을 선택해 바로 미니 코스를 받기 (Priority: P1)

주말이나 퇴근 후 무엇을 할지 정하지 못한 사용자는 오늘의 기분, 사용 가능 시간, 예산, 동행 여부를 선택한 뒤 자신에게 맞는 미니 코스 1개를 바로 확인한다.

**Why this priority**: 이 흐름이 앱의 핵심 가치다. 사용자가 고민 시간을 줄이고 바로 실행 가능한 선택지를 얻지 못하면 MVP가 성립하지 않는다.

**Independent Test**: 사용자는 네 가지 조건을 모두 선택하고 추천 요청을 실행해, 조건에 맞는 코스명과 활동 목록을 한 화면에서 확인할 수 있다.

**Acceptance Scenarios**:

1. **Given** 사용자가 첫 화면에 진입했고 아직 조건을 선택하지 않았을 때, **When** 기분, 시간, 예산, 동행 여부를 모두 선택하고 추천을 요청하면, **Then** 조건에 맞는 미니 코스 1개가 표시된다.
2. **Given** 선택한 조건에 맞는 코스가 여러 개 있을 때, **When** 사용자가 추천을 요청하면, **Then** 앱은 전체 예상 비용이 가장 낮은 코스 1개를 우선 표시하고, 비용이 같으면 전체 예상 시간이 더 짧은 코스를 표시하며, 왜 조건에 맞는지 짧게 보여준다.

---

### User Story 2 - 시간과 비용을 보고 실행 가능 여부 판단하기 (Priority: P2)

돈과 시간을 많이 쓰고 싶지 않은 사용자는 추천된 코스의 각 활동별 예상 시간과 예상 비용을 확인해 오늘 바로 실행 가능한지 판단한다.

**Why this priority**: Weekend Quest Planner는 복잡한 일정표가 아니라 빠른 결정을 돕는 앱이므로, 시간과 비용 정보가 결과의 신뢰도를 만든다.

**Independent Test**: 추천 결과를 받은 뒤 활동별 예상 시간, 활동별 예상 비용, 코스 전체 예상 시간과 전체 예상 비용이 모두 보이는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 추천 결과가 표시된 상태에서, **When** 사용자가 활동 목록을 확인하면, **Then** 각 활동은 이름, 예상 시간, 예상 비용을 함께 표시한다.
2. **Given** 추천 결과가 표시된 상태에서, **When** 사용자가 코스 요약을 확인하면, **Then** 전체 예상 시간은 선택한 사용 가능 시간을 넘지 않고 전체 예상 비용은 선택한 예산을 넘지 않는다.

---

### User Story 3 - 조건에 맞는 추천이 없을 때 이유를 이해하기 (Priority: P3)

조건을 너무 좁게 선택한 사용자는 추천이 없다는 사실만 보는 것이 아니라, 어떤 제약 때문에 결과가 없는지 이해하고 다시 선택할 수 있다.

**Why this priority**: 빈 결과는 실패처럼 느껴질 수 있다. 명확한 빈 상태 안내가 있어야 사용자가 흐름을 포기하지 않는다.

**Independent Test**: 의도적으로 매우 짧은 시간과 낮은 예산을 선택해 추천이 없는 상태를 만들고, 안내 문구와 다음 행동 힌트가 보이는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 선택한 조건에 맞는 코스가 없을 때, **When** 사용자가 추천을 요청하면, **Then** 앱은 "정확히 맞는 코스 없음" 빈 상태와 함께 시간 또는 예산 조건을 넓혀보라는 안내를 표시한다.
2. **Given** 빈 상태가 표시된 뒤, **When** 사용자가 조건을 다시 조정하고 추천을 요청하면, **Then** 새 조건을 기준으로 추천 또는 빈 상태가 다시 계산되어 표시된다.

---

### User Story 4 - 최근 추천 결과 다시 보기 (Priority: P3)

사용자는 같은 브라우저에서 앱을 다시 열었을 때 가장 최근에 받은 추천 결과를 확인해, 전에 고른 미니 코스를 다시 떠올릴 수 있다.

**Why this priority**: 추천 결과를 놓치지 않게 해주지만, 첫 추천 흐름보다 우선순위는 낮다.

**Independent Test**: 추천 결과를 받은 뒤 같은 브라우저에서 앱을 다시 열었을 때 최근 추천 영역에 마지막 결과가 표시되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 사용자가 추천 결과를 받은 적이 있을 때, **When** 같은 브라우저에서 앱을 다시 열면, **Then** 최근 추천 결과가 별도 영역에 표시된다.
2. **Given** 저장된 최근 추천 결과가 없을 때, **When** 사용자가 앱을 처음 열면, **Then** 최근 추천 영역은 비어 있거나 첫 추천을 유도하는 상태로 표시되고 추천 입력 흐름은 그대로 사용할 수 있다.
3. **Given** 사용자가 새 추천 결과를 받았을 때, **When** 같은 브라우저에서 앱을 다시 열면, **Then** 최근 추천 영역에는 가장 최근 1건의 선택 조건, 코스 요약, 활동별 예상 시간과 예상 비용이 표시된다.

### Edge Cases

- 선택 조건 중 하나라도 비어 있는 상태에서 추천을 요청하면, 누락된 조건을 선택하라는 안내가 표시된다.
- 선택한 조건을 모두 만족하는 샘플 코스가 없으면, "정확히 맞는 코스 없음" 빈 상태가 표시되고 시간 또는 예산을 넓혀보라는 다음 행동이 제안된다.
- 같은 기분과 동행 조건은 맞지만 시간 또는 예산을 초과하는 코스는 추천 결과로 표시되지 않는다.
- 최근 추천 결과를 읽을 수 없거나 사용할 수 없는 환경에서도 첫 화면과 새 추천 흐름은 계속 사용할 수 있다.
- 활동명이나 설명이 길어도 모바일과 데스크톱 화면에서 주요 정보가 겹치지 않아야 한다.

### Constitution Alignment *(mandatory)*

- **Beginner-readable scope**: 입력, 추천, 결과, 최근 결과라는 네 단계로 설명 가능한 첫 MVP 흐름에 한정한다.
- **Time and cost visibility**: 추천 코스는 활동별 예상 시간과 예상 비용을 반드시 보여주며, 코스 전체 예상 시간과 전체 예상 비용도 함께 보여준다.
- **Responsive behavior**: 조건 입력, 추천 요청, 결과 카드, 빈 상태, 최근 추천 영역은 모바일과 데스크톱에서 모두 읽고 조작할 수 있어야 한다.
- **Local-only boundary**: 추천은 샘플 데이터와 같은 브라우저의 최근 결과만 사용하며, 외부 서비스 호출, 계정 생성, 서버 저장은 포함하지 않는다.
- **Dependency boundary**: MVP 명세 단계에서는 새로운 외부 의존성을 요구하지 않는다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 사용자는 오늘의 기분을 4개의 고정 선택지 중 하나로 선택할 수 있어야 한다.
- **FR-002**: 사용자는 사용 가능 시간을 3개의 고정 선택지 중 하나로 선택할 수 있어야 한다.
- **FR-003**: 사용자는 사용할 예산 범위를 3개의 고정 선택지 중 하나로 선택할 수 있어야 한다.
- **FR-004**: 사용자는 혼자 갈지, 동행과 함께 갈지를 2개의 고정 선택지 중 하나로 선택할 수 있어야 한다.
- **FR-005**: 사용자가 필수 조건을 모두 선택하지 않고 추천을 요청하면, 앱은 어떤 조건이 빠졌는지 알려야 한다.
- **FR-006**: 앱은 선택한 기분, 시간, 예산, 동행 여부를 모두 만족하는 미니 코스만 추천해야 한다.
- **FR-007**: 추천 결과는 미니 코스 1개를 우선 제시해야 하며, 조건에 맞는 코스가 여러 개이면 전체 예상 비용이 가장 낮은 코스를 선택하고 비용이 같으면 전체 예상 시간이 더 짧은 코스를 선택해야 한다.
- **FR-008**: 추천 결과는 1개 이상의 활동으로 구성되어야 하며, 각 활동은 활동명, 예상 시간, 예상 비용을 표시해야 한다.
- **FR-009**: 추천 결과는 코스 전체 예상 시간과 전체 예상 비용을 표시해야 한다.
- **FR-010**: 코스 전체 예상 시간은 사용자가 선택한 사용 가능 시간을 넘지 않아야 한다.
- **FR-011**: 코스 전체 예상 비용은 사용자가 선택한 예산 범위를 넘지 않아야 한다.
- **FR-012**: 선택한 조건을 모두 만족하는 코스가 없으면, 앱은 "정확히 맞는 코스 없음" 빈 상태와 시간 또는 예산을 넓혀보라는 제안을 표시해야 한다.
- **FR-013**: 사용자가 새 조건으로 다시 추천을 요청하면, 이전 결과나 빈 상태는 새 조건에 맞는 결과로 대체되어야 한다.
- **FR-014**: 앱은 같은 브라우저에서 가장 최근 추천 결과 1건을 다시 볼 수 있게 해야 하며, 새 추천이 완료되면 이전 최근 결과를 새 결과로 교체해야 한다.
- **FR-015**: 최근 추천 결과는 사용자가 선택한 조건, 추천 코스 요약, 활동별 예상 시간, 활동별 예상 비용을 포함해야 한다.
- **FR-016**: 최근 추천 결과가 없거나 읽을 수 없을 때도 사용자는 새 추천 흐름을 정상적으로 시작할 수 있어야 한다.
- **FR-017**: 추천 흐름은 모바일과 데스크톱 화면에서 입력, 결과, 빈 상태의 핵심 정보가 겹치거나 잘리지 않아야 한다.

### Scope Boundaries

- 이 MVP는 오늘 또는 가까운 주말에 바로 실행할 수 있는 가벼운 미니 코스 추천에 한정한다.
- 일정 캘린더, 지도 길찾기, 예약, 결제, 사용자 계정, 공유 기능은 포함하지 않는다.
- 추천 데이터는 샘플 코스 목록에 있는 항목만 사용한다.
- 최근 추천 결과는 같은 브라우저에서만 다시 볼 수 있으며, 여러 기기 간 동기화는 포함하지 않는다.

### Key Entities *(include if feature involves data)*

- **User Preference**: 추천을 위해 사용자가 선택한 조건. 4개의 기분 선택지, 3개의 사용 가능 시간 선택지, 3개의 예산 범위 선택지, 2개의 동행 여부 선택지 중 사용자가 고른 값을 포함한다.
- **Mini Course**: 추천 가능한 하나의 코스. 코스명, 적합한 기분, 필요한 전체 시간, 필요한 전체 비용, 동행 적합성, 활동 목록을 포함한다.
- **Activity**: 미니 코스를 구성하는 개별 활동. 활동명, 예상 시간, 예상 비용을 포함한다.
- **Recommendation Result**: 사용자의 조건에 맞게 선택된 결과. 추천된 미니 코스, 조건에 맞는 이유, 전체 시간과 비용 요약을 포함한다.
- **Recent Recommendation**: 같은 브라우저에서 다시 보여줄 마지막 추천 1건. 선택 조건, 추천 코스 요약, 활동별 예상 시간과 예상 비용을 포함하며, 새 추천이 완료되면 최신 결과로 대체된다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 처음 방문한 사용자는 60초 안에 네 가지 조건을 선택하고 추천 결과 또는 빈 상태를 확인할 수 있다.
- **SC-002**: 대표 입력 조합 5개 이상에서 추천 결과 또는 빈 상태가 조건과 일치한다는 것을 검증할 수 있다.
- **SC-003**: 추천 결과가 표시될 때 사용자는 별도 화면 이동 없이 코스명, 활동별 예상 시간, 활동별 예상 비용, 전체 예상 시간, 전체 예상 비용을 확인할 수 있다.
- **SC-004**: 조건에 맞는 추천이 없을 때 사용자는 10초 안에 결과가 없는 이유와 다음에 조정할 수 있는 조건을 이해할 수 있다.
- **SC-005**: 같은 브라우저에서 앱을 다시 열었을 때, 이전에 추천을 받은 사용자는 최근 추천 결과를 확인할 수 있다.
- **SC-006**: 모바일과 데스크톱 화면 모두에서 주요 입력과 결과 텍스트가 겹치거나 잘리지 않고 읽힌다.

## Assumptions

- 사용자는 오늘 또는 이번 주말에 바로 실행할 수 있는 짧은 활동을 찾고 있다.
- 기분 선택지는 MVP에서 4개의 이해하기 쉬운 범주로 고정한다. 예: 쉬고 싶음, 움직이고 싶음, 새로움이 필요함, 사람과 시간을 보내고 싶음.
- 사용 가능 시간 선택지는 MVP에서 3개의 빠른 선택 범위로 고정한다. 예: 1시간 이하, 2시간 이하, 반나절 이하.
- 예산 선택지는 MVP에서 3개의 부담이 적은 범위로 고정한다. 예: 무료, 1만원 이하, 3만원 이하.
- 동행 여부는 혼자 또는 동행 있음의 2개 선택지로 고정하며, 둘 다 가능한 코스는 두 조건 모두에서 추천될 수 있다.
- 샘플 코스 목록은 추천 성공 사례와 빈 상태 사례를 모두 테스트할 수 있을 만큼 충분한 항목을 포함한다.
- 최근 추천 결과를 저장할 수 없는 환경에서는 저장 기능만 생략되고 새 추천 흐름은 계속 사용할 수 있다.
