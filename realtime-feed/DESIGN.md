# Realtime Guestbook Design

날짜: 2026-05-10
프로젝트: `realtime-feed`
문서 상태: UI/UX 디자인 방향 초안

## 1. Product Design Direction

Realtime Guestbook은 행사장 벽이나 수업 후 게시판에 붙은 포스트잇 방명록처럼 느껴져야 한다. 방문자는 로그인 없이 이름, 짧은 메시지, 무드, 아바타 색상, 선택 첨부만으로 빠르게 흔적을 남긴다. 피드는 새 글이 실시간으로 올라오는 작은 커뮤니티 공간처럼 살아 있어야 한다.

첫 화면은 마케팅 랜딩 페이지가 아니라 방명록 자체다. 데스크톱에서는 작성 영역과 피드 영역을 나누고, 모바일에서는 작성 영역을 먼저 보여준 뒤 피드를 자연스럽게 이어서 보여준다.

디자인 기준은 다음 세 가지다.

- 따뜻함: 종이, 손글씨, 작은 커뮤니티의 감각을 준다.
- 즉시성: 작성, 등록, 실시간 반영 상태가 분명하게 보인다.
- 가벼움: 소셜 앱처럼 복잡하지 않고, 누구나 바로 남길 수 있다.

## 2. Style Keywords

- Warm community
- Post-it gallery
- Handmade but tidy
- Realtime presence
- Friendly Korean UI
- Soft block layout
- Lightweight creative tool
- Mobile-first writing

피해야 할 방향:

- 차가운 SaaS 대시보드처럼 보이는 구성
- 제품 설명 중심의 마케팅 히어로
- 과한 유리 효과, 장식용 그라디언트, 오브 배경
- 카드 안에 카드가 반복되는 무거운 중첩 구조
- UI 아이콘을 이모지로 대체하는 방식

## 3. Color Palette

기본 배경은 따뜻한 종이색, 주요 액션은 코랄, 실시간 상태는 틸 계열을 사용한다. 포스트잇 카드는 여러 종이색을 쓰되 전체 화면이 한 가지 hue로만 읽히지 않게 한다.

| Role | Hex | Usage |
| --- | --- | --- |
| Background | `#FFF8ED` | 전체 앱 배경 |
| Surface | `#FFFFFF` | 작성 패널, 상세 패널, 입력 표면 |
| Paper Yellow | `#FFE8A3` | 기본 포스트잇 카드 |
| Paper Mint | `#CFF7E8` | 보조 포스트잇 카드 |
| Paper Peach | `#FFD6C2` | 보조 포스트잇 카드 |
| Paper Blue | `#D7ECFF` | 보조 포스트잇 카드 |
| Primary | `#F9735B` | 등록 버튼, 주요 액션 |
| Realtime | `#0F9F8F` | 연결됨, 새 글, 동기화 상태 |
| Accent | `#7C3AED` | 선택 상태, 작은 강조 |
| Text | `#1F2933` | 본문 텍스트 |
| Muted Text | `#5F6B7A` | 메타 정보, 보조 설명 |
| Border | `#E8DCCB` | 입력, 패널 경계 |
| Error | `#D92D20` | 오류 상태 |

아바타 색상은 PRD의 팔레트를 유지한다.

- `#2563eb`
- `#059669`
- `#d97706`
- `#dc2626`
- `#7c3aed`
- `#0f766e`

아바타 색상은 큰 배경색으로 쓰지 않고, 카드 상단의 작은 원형 스와치와 작성자 식별에 사용한다.

## 4. Typography Direction

기본 폰트는 `Noto Sans KR`을 사용한다. 한국어 UI의 읽기 흐름을 우선하며, 포스트잇 분위기는 폰트 자체보다 색, 표면, 간격, 카드 형태로 표현한다.

| Token | Size | Weight | Usage |
| --- | --- | --- | --- |
| Display | 28-32px | 700 | 앱 제목, 주요 섹션 제목 |
| Section Title | 20-24px | 700 | 작성 패널, 피드 제목 |
| Card Name | 15-16px | 700 | 작성자 이름 |
| Body | 15-16px | 400-500 | 메시지, 댓글 본문 |
| Metadata | 12-13px | 500 | 상대 시간, 댓글 수, 상태 |
| Button | 14-15px | 700 | 주요 버튼, 보조 버튼 |

권장 line-height:

- 본문: `1.55`
- 카드 메시지: `1.5`
- 메타 텍스트: `1.35`

텍스트는 부모 요소 안에서 줄바꿈되어야 하며, 긴 이름이나 긴 한국어 문장이 카드 밖으로 넘치지 않아야 한다.

## 5. Spacing And Radius Rules

| Token | Value |
| --- | --- |
| Mobile page padding | 16px |
| Tablet page padding | 24px |
| Desktop page padding | 32px |
| Main desktop gap | 28-32px |
| Mobile section gap | 16px |
| Panel padding | 20-24px |
| Form field gap | 12px |
| Card list gap | 14-16px |
| Feed vertical gap | 16px |
| Button height | 44-48px |
| Input radius | 10px |
| Panel radius | 14px |
| Post-it card radius | 6-8px |
| Detail surface radius | 16px |

포스트잇 카드는 살짝 손으로 붙인 듯한 느낌을 줄 수 있다. 다만 회전은 `-1deg`에서 `1deg` 사이로 제한하고, hover나 realtime highlight가 레이아웃 위치를 바꾸지 않게 한다.

## 6. Guestbook Form Pattern

작성 폼은 "짧고 완성감 있는 한 장의 메모 작성기"처럼 구성한다.

권장 순서:

1. 이름 입력
2. 무드 선택
3. 아바타 색상 선택
4. 메시지 입력
5. 첨부 방식 선택
6. 첨부 미리보기
7. 등록 버튼

폼 규칙:

- 모든 입력에는 보이는 label을 둔다.
- 오류 메시지는 해당 입력 바로 아래에 표시한다.
- 전역 오류만 상단에 모으지 않는다.
- 등록 중에는 버튼을 비활성화하고 진행 상태를 표시한다.
- 업로드 실패 시 메시지 입력값은 보존한다.
- 색상 선택은 원형 swatch를 사용한다.
- 무드 선택은 segmented control 또는 작은 chip 그룹을 사용한다.
- 첨부 방식은 이미지 업로드와 그리기 중 하나만 선택 가능해야 한다.

첨부 컨트롤:

- 이미지 업로드: 파일 선택, 파일명, 미리보기, 제거 버튼을 제공한다.
- 그리기: 펜 색상, 지우기, 초기화만 제공한다.
- 첨부가 없는 상태도 정상 흐름으로 다룬다.
- 잘못된 파일 타입이나 크기 초과는 업로드 전에 알려준다.

## 7. Post-it Card Gallery Pattern

피드는 최신 글이 위에 오는 포스트잇 갤러리다. 데스크톱에서는 작성 패널 오른쪽에 2-3열 카드 그리드를 배치하고, 모바일에서는 1열 목록으로 안정적으로 쌓는다.

카드 구성:

- 상단: 아바타 색상 원, 이름, 무드, 상대 시간
- 본문: 메시지
- 첨부: 이미지 또는 그림 미리보기
- 하단: 댓글 수, 펼치기 버튼, 본인 글일 때 삭제 버튼

카드 규칙:

- 카드 배경색은 `Paper Yellow`, `Paper Mint`, `Paper Peach`, `Paper Blue`를 순환 사용한다.
- 카드별 색상은 콘텐츠를 방해하지 않을 정도로 부드럽게 유지한다.
- 첨부 이미지는 고정 aspect-ratio 안에서 잘리지 않게 보여준다.
- 새로 들어온 글은 상단에 부드럽게 삽입하고 1-2초 정도 배경 하이라이트를 준다.
- 삭제 버튼은 현재 익명 세션이 작성한 글에만 보인다.
- hover 효과는 색, shadow, border 변화 중심으로 주고 scale로 레이아웃을 흔들지 않는다.

## 8. Detail And Comments UI Pattern

상세 화면은 별도 라우트보다 카드 확장, side panel, 또는 modal 패턴이 적합하다. 첫 버전에서는 구현 복잡도를 고려해 카드 확장형 댓글 영역을 기본 패턴으로 둔다.

데스크톱:

- 카드 펼치기 시 카드 아래에 댓글 영역을 확장한다.
- 추후 미디어를 크게 봐야 하는 흐름이 필요하면 오른쪽 side panel로 확장할 수 있다.

모바일:

- 카드 펼치기 시 카드 아래에 댓글 영역을 표시한다.
- 댓글이 길어지는 경우 bottom sheet 또는 full-screen detail로 전환할 수 있다.
- 닫기 또는 접기 컨트롤은 화면 상단/카드 하단에서 명확히 제공한다.

댓글 UI:

- 댓글은 작은 말풍선형 리스트로 보여준다.
- 각 댓글은 무드, 메시지, 상대 시간을 포함한다.
- 댓글 작성기는 한 줄 입력, 무드 선택, 전송 버튼으로 구성한다.
- 댓글 메시지는 240자 제한을 UI에서 보여준다.
- 펼쳐진 게시글에만 댓글 realtime 구독 상태를 표시한다.
- 본인 댓글에만 삭제 버튼을 보여준다.

## 9. Empty, Loading, Error, And Realtime States

### Empty

첫 글이 없을 때는 빈 흰 화면을 보여주지 않는다. 피드 영역에 첫 메시지를 유도하는 따뜻한 문구와 작성 폼으로 시선을 보내는 보조 안내를 둔다.

예시:

- "아직 남겨진 메모가 없어요. 첫 인사를 붙여 주세요."

### Loading

- 초기 피드 로딩은 포스트잇 카드 모양의 skeleton을 사용한다.
- 첨부 업로드 중에는 미리보기 영역 안에서 진행 상태를 보여준다.
- 댓글 로딩은 펼쳐진 댓글 영역 안에서만 표시한다.

### Error

- 입력 오류는 관련 필드 바로 아래에 표시한다.
- 피드 로드 실패는 피드 영역 안에 재시도 버튼과 함께 보여준다.
- 업로드 실패 시 게시글을 자동 생성하지 않는다.
- 삭제 실패는 해당 카드 또는 댓글 근처에 짧게 표시한다.
- 오류 문구는 짧은 한국어 문장으로 작성한다.

권장 문구:

- "방명록을 준비하지 못했어요. 잠시 후 다시 시도해 주세요."
- "첨부를 올리지 못했어요. 다시 시도하거나 첨부를 제거해 주세요."
- "내가 작성한 항목만 삭제할 수 있어요."
- "삭제하지 못했어요. 다시 시도해 주세요."

### Realtime Connected

- 작은 틸색 상태 점과 "실시간 연결됨" pill을 피드 상단에 둔다.
- 상태 표시가 주 콘텐츠보다 커지지 않게 한다.

### Realtime Disconnected

- 기존 피드는 유지한다.
- 피드 상단에 "실시간 연결을 다시 시도 중이에요." 상태를 표시한다.
- 재연결 후 최신 게시글과 펼쳐진 댓글을 조용히 다시 동기화한다.

## 10. Mobile Layout Priorities

모바일에서는 작성 경험이 먼저지만, 피드 읽기를 방해하지 않아야 한다.

우선순위:

1. 작성 폼은 첫 진입에서 바로 보이게 한다.
2. 등록 성공 후 작성 패널은 compact 상태로 줄인다.
3. 첨부 미리보기는 화면 너비를 넘지 않게 고정 비율을 둔다.
4. 포스트잇 카드는 1열로 안정적으로 쌓는다.
5. 댓글 영역은 펼친 카드 안에서 먼저 해결한다.
6. 댓글이 길어질 경우 bottom sheet 또는 full-screen detail을 고려한다.
7. 주요 버튼은 최소 44px 높이를 유지한다.
8. 이름, 메시지, 버튼 텍스트는 줄바꿈되며 부모 밖으로 넘치지 않는다.
9. 실시간 상태는 작은 pill로 유지하고 콘텐츠를 크게 밀어내지 않는다.
10. 작성 도중 키보드가 올라와도 등록 버튼과 오류 메시지를 찾을 수 있어야 한다.

## 11. Accessibility And Interaction Rules

- 모든 form input은 label과 연결한다.
- 오류 메시지는 `role="alert"` 또는 `aria-live` 영역에서 읽힐 수 있어야 한다.
- 색상만으로 상태를 구분하지 않는다.
- 모든 클릭 가능한 요소는 명확한 hover, active, focus-visible 상태를 가진다.
- 버튼과 컨트롤은 키보드로 접근 가능해야 한다.
- 이미지 첨부에는 alt 텍스트 정책을 둔다. 사용자가 입력하지 않는 경우 "작성자가 첨부한 이미지"처럼 안전한 대체 문구를 사용한다.
- 애니메이션은 150-300ms 범위로 유지한다.
- `prefers-reduced-motion` 사용자는 삽입/하이라이트 애니메이션을 줄인다.

## 12. Implementation Guardrails

이 문서는 디자인 방향 문서이며 React 코드를 포함하지 않는다.

구현 시 지켜야 할 기준:

- 첫 화면은 랜딩 페이지가 아니라 실제 방명록 앱이어야 한다.
- UI 컴포넌트에 Supabase 쿼리 로직을 직접 흩뿌리지 않는다.
- 작성, 첨부, 피드, 댓글, realtime 상태는 독립된 UI 상태로 드러나야 한다.
- 모바일 390px, 데스크톱 1280px에서 주요 입력, 버튼, 카드 텍스트가 넘치지 않아야 한다.
- 시각적 proof가 필요하면 데스크톱과 모바일 스크린샷으로 카드 밀도, 작성 폼, 댓글 확장 상태를 확인한다.
