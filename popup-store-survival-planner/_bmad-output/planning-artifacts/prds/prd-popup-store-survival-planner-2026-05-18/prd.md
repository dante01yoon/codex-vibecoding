# PRD: Pop-up Store Survival Planner

## 0. Document Purpose

이 PRD는 입문자 실습용 React 웹앱인 **Pop-up Store Survival Planner**의 MVP 범위와 요구사항을 정의한다. downstream 작업자는 이 문서를 기준으로 UX 설계, 아키텍처, 에픽/스토리 분해, 구현 계획을 만들 수 있다. 첫 강의용 MVP는 작은 클라이언트 전용 앱으로 제한하며, 로그인, 결제, 지도 API, 실시간 재고 조회, SNS 자동 공유, 서버 저장은 MVP에서 제외한다. 체크리스트 항목 직접 추가/삭제와 방문 후 메모는 첫 MVP가 아니라 선택 과제 또는 후속 범위로 둔다.

## 1. Vision

Pop-up Store Survival Planner는 인기 브랜드 팝업스토어에 가기 전 방문자가 자신의 일정, 예상 대기 시간, 굿즈 예산, 현장 체크리스트를 한 화면 흐름 안에서 정리하도록 돕는 웹앱이다.

팝업스토어 방문은 재미있는 경험이지만, 긴 대기 시간, 예산 초과, 현장 준비물 누락, 구매 우선순위 혼란 때문에 쉽게 지칠 수 있다. 이 앱은 방문 전 작은 계획을 세우게 하고, 현장에서 바로 확인할 수 있는 **오늘의 생존 플랜 카드**를 생성해 사용자가 더 차분하게 방문을 즐기도록 만든다.

MVP의 목적은 완성형 상용 서비스가 아니라, 입문자가 React의 controlled input, 리스트 렌더링, 조건부 렌더링, localStorage 저장을 자연스럽게 연습할 수 있는 작고 완결된 제품 경험을 만드는 것이다.

## 2. Target User

### 2.1 Primary Persona

**민지, 24세 팝업스토어 방문자**  
민지는 좋아하는 브랜드의 한정 팝업스토어에 방문하려고 한다. 굿즈를 사고 싶지만 예산을 넘기고 싶지 않고, 대기 시간이 길어도 현장에서 당황하지 않도록 준비물을 미리 확인하고 싶다.

### 2.2 Jobs To Be Done

- 인기 팝업스토어 방문 전 날짜와 시간대를 정리하고 싶다.
- 예상 대기 시간을 고려해 현장에서 지치지 않는 준비를 하고 싶다.
- 사고 싶은 굿즈의 우선순위와 예산을 미리 정해 충동구매를 줄이고 싶다.
- 방문 당일 확인할 체크리스트를 만들고 싶다.
- 방문 계획을 카드 형태로 저장해 나중에 다시 열어보고 싶다.
- 방문 후 간단한 기록을 남기고 싶다.

### 2.3 Non-Users for MVP

- 브랜드 운영자, 매장 관리자, 재고 관리 담당자.
- 팝업스토어 예약, 결제, 재고 확인을 앱 안에서 끝내려는 사용자.
- 여러 기기에서 같은 데이터를 동기화해야 하는 사용자.

### 2.4 Key User Journeys

- **UJ-1. 민지가 방문 전 생존 플랜을 만든다.**  
  민지는 방문 날짜와 시간대를 입력하고, 예상 대기 시간을 적는다. 사고 싶은 굿즈를 우선순위와 예산과 함께 입력한 뒤 현장 체크리스트를 확인한다. 앱은 입력값을 요약해 오늘의 생존 플랜 카드를 보여준다. 민지는 새로고침 후에도 계획이 남아 있는지 확인한다.

- **UJ-2. 민지가 현장에서 체크리스트를 보며 준비 상태를 확인한다.**  
  민지는 대기 중 휴대폰으로 저장된 플랜을 다시 열고 체크리스트 항목을 하나씩 체크한다. 체크 상태가 즉시 반영되고 저장되어, 화면을 닫았다가 다시 열어도 현재 준비 상태를 이어서 볼 수 있다.

- **UJ-3. 민지가 방문 후 간단한 기록을 남긴다.** *(선택 과제)*  
  방문이 끝난 뒤 민지는 오늘의 플랜에 한 줄 메모를 남긴다. 이 흐름은 첫 강의용 MVP가 아니라 후속 선택 과제로 둔다. [ASSUMPTION: 방문 후 기록을 추가할 경우 사진, SNS 공유, 상세 일지가 아니라 localStorage에 저장되는 짧은 텍스트 메모로 제한한다.]

## 3. Glossary

- **방문 계획** — 방문 날짜, 시간대, 예상 대기 시간, 굿즈 계획, 체크리스트, 생존 플랜 카드를 포함하는 하나의 로컬 데이터 묶음.
- **방문 날짜** — 사용자가 팝업스토어에 방문하려는 날짜.
- **시간대** — 오전, 오후, 저녁 또는 사용자가 직접 입력한 방문 예상 시간 범위.
- **예상 대기 시간** — 사용자가 예상하는 현장 대기 시간. 분 단위 숫자 또는 간단한 구간으로 입력한다.
- **굿즈 항목** — 사용자가 구매를 고려하는 상품. 이름, 우선순위, 예상 예산을 가진다.
- **우선순위** — 굿즈 항목의 중요도. MVP에서는 높음, 보통, 낮음으로 제한한다.
- **현장 체크리스트** — 방문 당일 준비물과 행동을 확인하기 위한 체크 가능한 항목 목록.
- **생존 플랜 카드** — 방문 계획의 핵심 정보를 한 화면에 요약한 카드.
- **방문 후 메모** — 방문이 끝난 뒤 사용자가 남기는 짧은 텍스트 기록. 첫 MVP가 아니라 선택 과제로 다룬다.
- **로컬 저장** — 서버 없이 사용자의 브라우저 localStorage에 방문 계획을 저장하는 방식.

## 4. Product Shape

MVP는 단일 페이지 웹앱으로 구성한다. 사용자는 위에서 아래로 입력을 진행하고, 같은 화면 하단 또는 옆 영역에서 생존 플랜 카드를 확인한다.

주요 화면 영역:

- 방문 정보 입력 영역.
- 굿즈 예산 입력 영역.
- 현장 체크리스트 영역.
- 오늘의 생존 플랜 카드 영역.
- 저장 상태와 초기화 액션 영역.

## 5. Features

### 5.1 방문 정보 입력

**Description:** 사용자는 팝업스토어 방문 날짜, 시간대, 예상 대기 시간을 입력한다. 이 정보는 생존 플랜 카드의 기본 요약으로 사용된다. Realizes UJ-1.

#### FR-1: 방문 날짜 입력

사용자는 방문 날짜를 입력할 수 있다.

**Consequences:**

- 날짜 입력값이 비어 있으면 생존 플랜 카드에 날짜 미정 상태가 표시된다.
- 날짜 입력값이 있으면 생존 플랜 카드에 같은 날짜가 표시된다.
- 날짜 입력은 localStorage 저장 대상에 포함된다.

#### FR-2: 방문 시간대 입력

사용자는 방문 시간대를 입력할 수 있다.

**Consequences:**

- MVP는 오전, 오후, 저녁 중 하나를 고르거나 짧은 텍스트로 직접 입력하는 방식을 허용한다. [ASSUMPTION: 입문자 실습 난이도를 낮추기 위해 복잡한 시간 선택기는 사용하지 않는다.]
- 시간대 입력값은 생존 플랜 카드에 표시된다.
- 시간대 입력은 localStorage 저장 대상에 포함된다.

#### FR-3: 예상 대기 시간 입력

사용자는 예상 대기 시간을 입력할 수 있다.

**Consequences:**

- 예상 대기 시간은 분 단위 숫자로 입력한다.
- 값이 0보다 작거나 숫자가 아니면 사용자에게 유효하지 않은 입력임을 보여준다.
- 120분 이상이면 생존 플랜 카드에 긴 대기 대비 메시지를 표시한다.
- 예상 대기 시간은 localStorage 저장 대상에 포함된다.

### 5.2 굿즈 우선순위와 예산 입력

**Description:** 사용자는 구매하고 싶은 굿즈 항목을 추가하고, 각 항목의 우선순위와 예상 예산을 입력한다. 앱은 전체 예상 예산을 계산해 보여준다. Realizes UJ-1.

#### FR-4: 굿즈 항목 추가와 삭제

사용자는 굿즈 항목을 추가하고 삭제할 수 있다.

**Consequences:**

- 각 굿즈 항목은 이름, 우선순위, 예상 예산을 가진다.
- 이름이 비어 있는 항목은 생존 플랜 카드의 우선 구매 목록에 표시하지 않는다.
- 사용자는 최소 0개, 최대 5개의 굿즈 항목을 관리할 수 있다. [ASSUMPTION: MVP는 학습용 범위를 유지하기 위해 항목 수를 5개로 제한한다.]

#### FR-5: 굿즈 우선순위 지정

사용자는 각 굿즈 항목의 우선순위를 높음, 보통, 낮음 중 하나로 지정할 수 있다.

**Consequences:**

- 생존 플랜 카드는 우선순위가 높은 굿즈를 먼저 보여준다.
- 같은 우선순위 안에서는 사용자가 입력한 순서를 유지한다.
- 우선순위 값은 localStorage 저장 대상에 포함된다.

#### FR-6: 굿즈 예산 계산

사용자는 각 굿즈 항목의 예상 예산을 입력하고 전체 예상 예산을 확인할 수 있다.

**Consequences:**

- 각 항목의 예상 예산은 0 이상의 숫자로 입력한다.
- 앱은 모든 굿즈 항목의 예상 예산 합계를 계산한다.
- 예산 합계는 생존 플랜 카드에 표시된다.
- MVP는 실제 결제, 가격 비교, 실시간 재고 정보를 제공하지 않는다.

### 5.3 현장 체크리스트

**Description:** 사용자는 팝업스토어 현장에서 확인할 기본 체크리스트를 볼 수 있고, 항목을 체크하거나 체크 해제할 수 있다. 직접 항목 추가와 삭제는 첫 MVP가 아니라 선택 과제로 둔다. Realizes UJ-1, UJ-2.

#### FR-7: 기본 체크리스트 제공

앱은 처음 사용할 때 기본 현장 체크리스트를 제공한다.

**Consequences:**

- 기본 항목은 보조 배터리, 물, 입장 정보 확인, 카드/현금 준비 확인, 굿즈 보관 가방을 포함한다.
- 사용자는 각 항목을 체크하거나 체크 해제할 수 있다.
- 체크 상태는 localStorage 저장 대상에 포함된다.

#### FR-8: 체크리스트 항목 추가와 삭제 *(선택 과제)*

사용자는 직접 체크리스트 항목을 추가하고 삭제할 수 있다. 이 요구사항은 첫 강의용 MVP가 아니라 선택 과제 또는 후속 범위로 둔다.

**Consequences:**

- 빈 텍스트 항목은 추가되지 않는다.
- 삭제된 항목은 저장된 방문 계획에서도 제거된다.
- 체크리스트가 모두 체크되면 생존 플랜 카드에 준비 완료 메시지를 표시한다.

### 5.4 오늘의 생존 플랜 카드 생성

**Description:** 앱은 방문 정보, 예상 대기 시간, 굿즈 우선순위, 예산 합계, 체크리스트 진행 상태를 모아 오늘의 생존 플랜 카드로 보여준다. Realizes UJ-1, UJ-2.

#### FR-9: 생존 플랜 카드 요약 표시

앱은 사용자의 현재 입력값을 기반으로 생존 플랜 카드를 자동 생성한다.

**Consequences:**

- 카드는 방문 날짜, 시간대, 예상 대기 시간, 굿즈 예산 합계, 우선 구매 굿즈, 체크리스트 진행 상태를 표시한다.
- 입력값이 변경되면 카드는 즉시 업데이트된다.
- 필수 입력값이 비어 있어도 카드는 깨지지 않고 미정 상태를 표시한다.

#### FR-10: 상황별 생존 메시지 표시

앱은 예상 대기 시간과 체크리스트 상태를 바탕으로 짧은 안내 메시지를 표시한다.

**Consequences:**

- 예상 대기 시간이 120분 이상이면 긴 대기 대비 메시지를 표시한다.
- 체크리스트 완료율이 100%이면 준비 완료 메시지를 표시한다.
- MVP의 메시지는 규칙 기반 텍스트이며 AI 생성 문구를 사용하지 않는다. [ASSUMPTION: 입문자 실습 앱이므로 외부 AI API를 붙이지 않는다.]

### 5.5 로컬 저장과 초기화

**Description:** 방문 계획은 서버 없이 사용자의 브라우저에 저장된다. 사용자는 앱을 닫거나 새로고침해도 최근 방문 계획을 다시 볼 수 있다. Realizes UJ-1, UJ-2.

#### FR-11: localStorage 기반 자동 저장

앱은 방문 계획의 변경 내용을 localStorage에 저장한다.

**Consequences:**

- 저장 대상은 방문 날짜, 시간대, 예상 대기 시간, 굿즈 항목, 체크리스트를 포함한다.
- 사용자가 브라우저를 새로고침하면 마지막 저장 상태가 복원된다.
- localStorage를 사용할 수 없는 환경에서는 저장 실패를 사용자에게 짧게 알린다.

#### FR-12: 방문 계획 초기화

사용자는 저장된 방문 계획을 초기화할 수 있다.

**Consequences:**

- 초기화 액션은 사용자 확인 후 실행한다.
- 초기화 후 입력값은 기본 상태로 돌아간다.
- 초기화 후 localStorage에 저장된 방문 계획도 삭제된다.

#### FR-13: 방문 후 메모 저장 *(선택 과제)*

사용자는 방문 후 짧은 텍스트 메모를 남길 수 있다. 이 요구사항은 첫 강의용 MVP가 아니라 선택 과제 또는 후속 범위로 둔다.

**Consequences:**

- 방문 후 메모는 선택 입력이다.
- 방문 후 메모는 생존 플랜 카드 또는 별도 메모 영역에 표시된다.
- 방문 후 메모는 localStorage 저장 대상에 포함된다.
- 후속 구현에서도 사진 첨부, SNS 공유, 타임라인 기록은 제공하지 않는다.

## 6. Cross-Cutting NFRs

- **Usability:** 첫 사용자가 별도 설명 없이 3분 안에 방문 계획을 만들 수 있어야 한다.
- **Accessibility:** 모든 입력은 연결된 label을 가져야 하며, 체크박스와 버튼은 키보드로 조작 가능해야 한다.
- **Responsive Layout:** 모바일 폭에서 입력 영역과 생존 플랜 카드가 세로로 자연스럽게 쌓여야 한다.
- **Privacy:** MVP는 서버로 데이터를 전송하지 않는다. 모든 데이터는 사용자 브라우저에만 저장된다.
- **Reliability:** 새로고침 후에도 localStorage에 저장된 마지막 방문 계획이 복원되어야 한다.
- **Beginner Implementation Fit:** 기능은 React 입문자가 `useState`, controlled input, list rendering, conditional rendering, `useEffect`, localStorage를 연습할 수 있는 수준으로 유지한다.

## 7. Non-Goals

- 로그인 또는 계정 시스템을 제공하지 않는다.
- 결제 기능을 제공하지 않는다.
- 지도 API 또는 위치 기반 길찾기를 제공하지 않는다.
- 실시간 재고 조회를 제공하지 않는다.
- SNS 자동 공유를 제공하지 않는다.
- 서버 저장 또는 여러 기기 간 동기화를 제공하지 않는다.
- 브랜드 운영자를 위한 관리자 기능을 제공하지 않는다.
- 실제 팝업스토어 예약 기능을 제공하지 않는다.

## 8. MVP Scope

### 8.1 In Scope

- 방문 날짜 입력.
- 방문 시간대 입력.
- 예상 대기 시간 입력.
- 굿즈 항목, 우선순위, 예상 예산 입력.
- 전체 굿즈 예상 예산 계산.
- 기본 현장 체크리스트.
- 오늘의 생존 플랜 카드 자동 생성.
- localStorage 기반 자동 저장과 복원.
- 저장된 방문 계획 초기화.

### 8.2 Out of Scope for MVP

- 로그인: 입문자 실습 범위를 넘기고 서버/인증 설계를 요구한다.
- 결제: 실제 구매 흐름과 보안 요구사항이 생긴다.
- 지도 API: 외부 API 키, 위치 권한, 에러 처리가 추가된다.
- 실시간 재고 조회: 브랜드별 재고 데이터 연동이 필요하다.
- SNS 자동 공유: 외부 플랫폼 연동과 권한 처리가 필요하다.
- 서버 저장: 백엔드, 데이터베이스, 배포 범위가 추가된다.
- 다중 방문 계획 관리: MVP는 최근 방문 계획 1개를 저장한다. [ASSUMPTION: 입문자 실습 범위를 위해 여러 팝업스토어 일정 관리는 v2로 미룬다.]
- 사용자 정의 체크리스트 항목 추가와 삭제: 첫 강의용 MVP 이후 선택 과제로 둔다.
- 방문 후 짧은 텍스트 메모: 첫 강의용 MVP 이후 선택 과제로 둔다.

## 9. Success Metrics

**Primary**

- **SM-1:** 사용자는 3분 안에 방문 계획을 완성하고 생존 플랜 카드를 볼 수 있다. Validates FR-1 through FR-7, FR-9, FR-10.
- **SM-2:** 사용자가 새로고침해도 마지막 방문 계획이 복원된다. Validates FR-11.
- **SM-3:** 입문자는 첫 MVP 구현을 통해 controlled input, list rendering, conditional rendering, localStorage 저장을 한 번씩 사용한다. Validates FR-1 through FR-7, FR-9 through FR-12.

**Secondary**

- **SM-4:** 사용자는 굿즈 예산 합계를 보고 구매 우선순위를 조정할 수 있다. Validates FR-4 through FR-6.
- **SM-5:** 사용자는 현장에서 기본 체크리스트를 열고 체크 상태를 이어서 관리할 수 있다. Validates FR-7, FR-11.

**Counter-metrics**

- **SM-C1:** 기능 수를 늘리는 것을 성공으로 보지 않는다. MVP는 입문자가 이해 가능한 작은 앱이어야 한다.
- **SM-C2:** 외부 API나 서버 연동을 붙이는 것을 성공으로 보지 않는다. MVP의 학습 목표는 클라이언트 상태 관리와 로컬 저장이다.
